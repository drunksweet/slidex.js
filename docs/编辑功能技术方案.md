## 1. 问题定义

tang-slidex 的 slide 文件是**纯 HTML**。编辑的本质问题是：

> **浏览器里用户对渲染结果的操作，如何精确、无损地映射回对应 HTML 文件的源码？**

### 1.1 当前方案的根本局限

现有 `EditManager` 使用 `nth-child` CSS selector + JSDOM 序列化的方案：

```
浏览器操作 → patches[{ type, selector, value }] → POST /api/save-slide
          → JSDOM 解析 HTML → querySelector → 修改 DOM → dom.serialize()
```

**四个核心缺陷：**

| 缺陷 | 说明 |
|------|------|
| **Selector 脆弱** | `#slide-host :nth-child(3) > :nth-child(2)` 这类路径只要 HTML 结构稍变就失效 |
| **格式全损** | `JSDOM.serialize()` 会重排所有 HTML——注释丢失、缩进变化、自闭合标签被修改 |
| **无行号概念** | 不知道某个元素对应文件第几行，无法精确定位和替换 |
| **AI 不可用** | 没有上下文信息，无法告诉 Agent"用户在操作哪里" |

### 1.2 核心思路：构建时注入行号（借鉴 code-inspector）

code-inspector（[github.com/zh-lx/code-inspector](https://github.com/zh-lx/code-inspector)）的实现原理是：

```
构建时（Babel/Vite Plugin）
  → 解析 JSX/HTML AST
  → 给每个元素注入 data 属性：
      data-inspector-line="5"
      data-inspector-filename="/src/App.tsx"

运行时
  → 点击元素 → 读取 data 属性 → 知道精确的文件 + 行号
  → POST devServer → 打开 IDE 定位到对应行
```

**针对 tang-slidex 的适配：**  
slide HTML 不是 JSX，但同样可以在 **Vite transform 阶段** 解析 HTML，给每个元素注入 `data-tang-line` 属性。这是整个编辑系统升级的基石。

## 2. 两种编辑路线概述

```
┌──────────────────────────────────────────────────────────────────┐
│                         用户的编辑意图                            │
└──────────────┬────────────────────────┬─────────────────────────┘
               │                        │
     直接操作 UI                    语言描述需求
               │                        │
┌──────────────▼──────────┐   ┌─────────▼────────────────────────┐
│   路线一：WYSIWYG 编辑  │   │   路线二：AI Agent 辅助编辑      │
│                         │   │                                   │
│  双击文字 → 改文字       │   │  选中区域 → 进入 AI 对话         │
│  拖拽元素 → 改位置       │   │  自然语言描述 → Agent 修改代码   │
│  调整样式 → 改 CSS       │   │  精确定位 → 行级代码更新         │
│                         │   │                                   │
│  用户无需感知代码        │   │  用户无需感知代码                 │
│  操作立即反馈            │   │  AI 处理复杂的逻辑/样式修改       │
└──────────────┬──────────┘   └─────────┬────────────────────────┘
               │                        │
               └───────────┬────────────┘
                            │
              ┌─────────────▼──────────────┐
              │  行号 + 文件路径（精确锚点）  │
              │  data-tang-line / file       │
              └─────────────┬──────────────┘
                            │
              ┌─────────────▼──────────────┐
              │  精确回写 HTML 源文件        │
              │  （行级替换，零格式损耗）     │
              └─────────────┬──────────────┘
                            │
              ┌─────────────▼──────────────┐
              │      Vite HMR 热更新         │
              └────────────────────────────┘
```

两种路线的**关键共同点**：都依赖同一套行号注入机制，都最终落到精确的源码修改。区别只在于**"谁来生成修改内容"**——路线一是用户直接操作，路线二是 AI Agent 生成。

## 3. 路线一：可视化直接编辑（WYSIWYG）

### 3.1 现有方案的缺陷分析

当前 `_getSelector` 生成 `#slide-host :nth-child(3) > :nth-child(2)` 这类脆弱路径，服务端用 JSDOM 序列化写回，格式全损。

**需要解决的核心问题：**
1. 如何稳定地定位一个 DOM 元素对应源码的哪一行？
2. 如何修改文本/样式后，精确地只改那一行，不破坏其他内容？

### 3.2 核心改进：行号注入机制

#### 3.2.1 Vite transform 插件

在 Vite 加载 slide HTML 时，解析 HTML 并给每个元素注入行号信息（仅开发模式）：

```javascript
// vite.config.js 新增 plugin：slideInspectorPlugin()

function slideInspectorPlugin() {
  return {
    name: 'tang-slidex-inspector',
    // 拦截 slides/*.html 的加载
    transform(code, id) {
      if (!id.match(/slides\/slide-\d+\.html$/)) return null
      if (process.env.NODE_ENV !== 'development') return null

      return injectLineNumbers(code, id)
    }
  }
}

/**
 * 解析 HTML，给每个元素注入 data-tang-line 和 data-tang-file
 * 使用 htmlparser2（轻量、保留格式）
 */
function injectLineNumbers(html, filePath) {
  // 逐行扫描，找到每个开标签，记录行号
  // 注入 data-tang-line="N" data-tang-file="slides/slide-001.html"
  const lines = html.split('\n')
  const result = lines.map((line, i) => {
    // 匹配 HTML 开标签，不含 void elements（br/hr/img/input 等）
    return line.replace(
      /<([a-zA-Z][a-zA-Z0-9]*)(\s[^>]*)?(>)/g,
      (match, tag, attrs = '', close) => {
        // 跳过 style/script/meta/link 等不可编辑标签
        if (['style','script','meta','link','head','html','body'].includes(tag.toLowerCase())) {
          return match
        }
        const lineAttr = ` data-tang-line="${i + 1}"`
        const fileAttr = ` data-tang-file="${path.relative(process.cwd(), filePath)}"`
        return `<${tag}${attrs}${lineAttr}${fileAttr}${close}`
      }
    )
  })
  return result.join('\n')
}
```

**注入效果：**
```html
<!-- 原始 slide-001.html 第 10 行 -->
<h1 class="slide-title">tang-slidex</h1>

<!-- Vite transform 后（仅开发模式，不写回文件） -->
<h1 class="slide-title" data-tang-line="10" data-tang-file="slides/slide-001.html">tang-slidex</h1>
```

> **关键**：注入只在内存中发生，不修改源文件。`data-tang-line` 是开发时的"元数据标记"。

#### 3.2.2 EditManager 升级：用行号替代 nth-child

浏览器端的 `EditManager` 修改获取锚点的方式：

```javascript
// 升级后的 _getAnchor()，替换现有 _getSelector()
_getAnchor(el) {
  // 优先使用行号信息（精确）
  const line = el.dataset.tangLine
  const file = el.dataset.tangFile
  if (line && file) {
    return { type: 'line', file, line: parseInt(line) }
  }
  // fallback：如果没有行号（极少情况），退回到 selector
  return { type: 'selector', selector: this._getSelector(el) }
}
```

**patch 数据结构升级：**
```typescript
type Patch =
  | {
      type: 'text'
      anchor: { type: 'line'; file: string; line: number }
      value: string         // 新文本内容
      originalValue: string // 原始内容（用于冲突检测）
    }
  | {
      type: 'move'
      anchor: { type: 'line'; file: string; line: number }
      dx: number; dy: number
    }
  | {
      type: 'style'
      anchor: { type: 'line'; file: string; line: number }
      property: string; value: string  // 具体 CSS 属性
    }
  | {
      type: 'delete'
      anchor: { type: 'line'; file: string; line: number }
    }
```

### 3.3 服务端精确写回方案

**彻底废弃 JSDOM**，改用基于行的字符串操作：

```javascript
// vite.config.js - applyPatchesByLine()（替换现有 applyPatches）

function applyPatchesByLine(html, patches) {
  const lines = html.split('\n')

  for (const patch of patches) {
    if (patch.anchor.type !== 'line') {
      // fallback 到 JSDOM（保留兼容性）
      continue
    }
    
    const lineIndex = patch.anchor.line - 1  // 0-based
    const line = lines[lineIndex]
    if (line === undefined) continue

    switch (patch.type) {
      case 'text': {
        // 只替换标签内的文本内容，保留标签和属性
        // 例：<h1 class="title" data-tang-line="10">旧文本</h1>
        //  → <h1 class="title" data-tang-line="10">新文本</h1>
        lines[lineIndex] = line.replace(
          /(>[^<]*)(<)/,
          (_, _open, close) => `>${patch.value}${close}`
        )
        break
      }
      
      case 'move': {
        // 在行内的 style 属性中累加 transform: translate(dx, dy)
        // 如果没有 style 属性，则新增
        const existing = line.match(/style="([^"]*)"/)
        if (existing) {
          const newStyle = mergeTranslate(existing[1], patch.dx, patch.dy)
          lines[lineIndex] = line.replace(/style="[^"]*"/, `style="${newStyle}"`)
        } else {
          lines[lineIndex] = line.replace(
            /(>)/,
            ` style="transform: translate(${patch.dx}px, ${patch.dy}px)"$1`
          )
        }
        break
      }

      case 'style': {
        // 修改内联 style 的某个属性
        const existing = line.match(/style="([^"]*)"/)
        if (existing) {
          const newStyle = setStyleProperty(existing[1], patch.property, patch.value)
          lines[lineIndex] = line.replace(/style="[^"]*"/, `style="${newStyle}"`)
        } else {
          lines[lineIndex] = line.replace(
            /(>)/,
            ` style="${patch.property}: ${patch.value}"$1`
          )
        }
        break
      }

      case 'delete': {
        // 将当前行及其对应的闭合标签行标记为删除
        // 简单方案：替换为 HTML 注释（可恢复）
        // 完整方案：找到对应闭合标签，删除整个块
        lines[lineIndex] = `<!-- [tang-deleted] ${lines[lineIndex].trim()} -->`
        break
      }
    }
  }

  // 移除 data-tang-* 属性后写回（避免污染源文件）
  const cleaned = lines.join('\n')
    .replace(/\s*data-tang-line="\d+"(\s*)/g, '$1')
    .replace(/\s*data-tang-file="[^"]*"(\s*)/g, '$1')

  return cleaned
}
```

**行级替换 vs JSDOM 对比：**

| 特性 | JSDOM 序列化 | 行级字符串替换 |
|------|-------------|-------------|
| 原始格式保留 | ❌ 全部重排 | ✅ 只改目标行 |
| HTML 注释保留 | ❌ 丢失 | ✅ 保留 |
| 缩进/空白保留 | ❌ 重排 | ✅ 保留 |
| 复杂嵌套结构 | ✅ 可处理 | ⚠️ 需额外处理多行元素 |
| 安全性（注入防御） | ✅ JSDOM 会转义 | ⚠️ 需手动转义 |

### 3.4 完整数据流

```
用户双击 slide 中的标题
  │
  ├─ 1. 读取 el.dataset.tangLine = "10"，el.dataset.tangFile = "slides/slide-001.html"
  │
  ├─ 2. 用户编辑 → blur 事件
  │      → patch: { type:'text', anchor:{type:'line',file:'slides/slide-001.html',line:10}, value:'新标题' }
  │
  ├─ 3. Ctrl+S → POST /api/save-slide
  │      body: { slideIndex:0, patches:[...] }
  │
  ├─ 4. Vite slideSavePlugin（升级版）
  │      → 读取文件
  │      → applyPatchesByLine(html, patches)
  │      → 写回文件（仅第 10 行被修改）
  │
  └─ 5. Vite HMR → tang-slide-update → 浏览器重载当前页 → 行号重新注入
```

## 4. 路线二：AI Agent 辅助编辑

### 4.1 核心思路：选区 = 上下文信号

**用户选中 slide 上的某个区域（或点击某个元素），本质上是在告诉 Agent：**

> "我接下来的指令，是针对这个区域/元素的"

这是一种**精确的上下文信号**，解决了 AI 对话中"你说的是哪里"的歧义问题。

**与路线一的本质区别：**

| 维度 | 路线一（WYSIWYG） | 路线二（AI 辅助） |
|------|-----------------|----------------|
| 适合的操作 | 简单的文字修改、位置调整 | 复杂的重排版、样式优化、内容扩写 |
| 用户感知 | 所见即所得，立即反馈 | 描述意图，AI 处理 |
| 技术路径 | DOM 操作 → 行级回写 | 选区 → 上下文 → Agent → 代码修改 |
| 代码修改精度 | 行级（有限） | 任意范围（AI 生成完整 HTML） |

### 4.2 选区系统设计

#### 4.2.1 选区的两种方式

**方式 A：元素级选中（精确）**
- 用户**单击**某个块级元素（与现有 WYSIWYG 编辑模式的"选中框"类似）
- 选中后，该元素的完整 HTML 片段（含子节点）作为上下文

**方式 B：范围框选（灵活）**
- 用户在 slide 上**拖拽框选**一个矩形区域
- 框选范围内的所有元素（通过 `getBoundingClientRect` 计算）作为上下文
- 类似 Figma 的框选行为

#### 4.2.2 SelectionManager 设计

```javascript
class SelectionManager {
  constructor() {
    this.selectedElements = []     // 当前选中的元素列表
    this.mode = 'none'             // 'none' | 'element' | 'range'
  }

  /**
   * 单元素选中
   * @param {HTMLElement} el
   */
  selectElement(el) {
    this.selectedElements = [el]
    this.mode = 'element'
    this._highlight()
    this._emit()
  }

  /**
   * 框选范围内的元素
   * @param {DOMRect} rect  框选区域的 viewport 坐标
   */
  selectByRect(rect) {
    const slide = document.querySelector('.slide')
    if (!slide) return
    
    this.selectedElements = Array.from(slide.querySelectorAll('*'))
      .filter(el => {
        // 排除装饰性元素
        if (['STYLE','SCRIPT','CANVAS'].includes(el.tagName)) return false
        const r = el.getBoundingClientRect()
        // 元素矩形与框选矩形有交叉
        return !(r.right < rect.left || r.left > rect.right ||
                 r.bottom < rect.top || r.top > rect.bottom)
      })
    this.mode = 'range'
    this._highlight()
    this._emit()
  }

  /**
   * 将选中内容序列化为 Agent 可用的上下文
   */
  toAgentContext() {
    const fragments = this.selectedElements.map(el => {
      const line = el.dataset.tangLine
      const file = el.dataset.tangFile
      // 获取原始 HTML（去除 data-tang-* 属性）
      const outerHTML = el.outerHTML
        .replace(/\s*data-tang-line="\d+"(\s*)/g, '$1')
        .replace(/\s*data-tang-file="[^"]*"(\s*)/g, '$1')
      return {
        tagName: el.tagName.toLowerCase(),
        classList: [...el.classList].join(' '),
        line: line ? parseInt(line) : null,
        file,
        html: outerHTML,
        textContent: el.textContent.trim()
      }
    })

    return {
      slideFile: `slides/slide-${String(currentSlide + 1).padStart(3,'0')}.html`,
      slideIndex: currentSlide,
      selectionMode: this.mode,
      selectedCount: this.selectedElements.length,
      fragments,
      // 整个 slide 的完整 HTML（作为更宽的上下文）
      fullSlideHtml: document.querySelector('.slide')?.outerHTML ?? ''
    }
  }

  /** 清除选中 */
  clear() {
    this.selectedElements = []
    this.mode = 'none'
    this._clearHighlight()
    this._emit()
  }

  _highlight() { /* 渲染蓝色高亮框 */ }
  _clearHighlight() { /* 移除高亮 */ }
  _emit() {
    // 通知 AI 对话框：上下文已更新
    window.dispatchEvent(new CustomEvent('tang:selection-change', {
      detail: this.toAgentContext()
    }))
  }
}
```

### 4.3 Agent 接口协议

#### 4.3.1 发送给 Agent 的上下文结构

```typescript
interface AgentEditContext {
  // 当前操作的 slide
  slideFile: string        // "slides/slide-001.html"
  slideIndex: number       // 0-based
  
  // 用户选中的区域
  selectionMode: 'element' | 'range' | 'none'
  selectedCount: number
  fragments: Array<{
    tagName: string          // "h1"
    classList: string        // "slide-title animate-slide-up"
    line: number | null      // 源文件行号（精确定位锚点）
    file: string             // "slides/slide-001.html"
    html: string             // 完整的 HTML 片段
    textContent: string      // 纯文本（方便理解语义）
  }>
  
  // 完整 slide HTML（宽上下文）
  fullSlideHtml: string
  
  // 用户的自然语言指令（由对话框附加）
  userInstruction: string
}
```

#### 4.3.2 Agent 返回的修改指令

Agent 看到上下文后，生成**结构化的修改指令**（不是自由文本），这样前端可以精确执行：

```typescript
interface AgentEditResponse {
  // 修改类型
  editType: 'patch' | 'replace-element' | 'replace-slide'
  
  // 当 editType = 'patch'：精确的行级修改
  patches?: Array<{
    line: number
    file: string
    action: 'replace-line' | 'replace-text' | 'insert-after' | 'delete'
    newContent?: string   // 新的行内容（replace-line 时用）
    newText?: string      // 新的文本内容（replace-text 时用）
  }>
  
  // 当 editType = 'replace-element'：替换整个元素的 HTML
  elementReplacements?: Array<{
    line: number
    file: string
    newHtml: string   // 完整的新 HTML 片段
  }>
  
  // 当 editType = 'replace-slide'：重写整个 slide
  newSlideHtml?: string
  
  // Agent 的说明（展示给用户）
  explanation: string
}
```

#### 4.3.3 服务端 AI 端点

```javascript
// Vite 插件新增端点：POST /api/ai-edit

server.middlewares.use('/api/ai-edit', async (req, res) => {
  const { context, userInstruction } = await parseBody(req)
  
  // 构造发给 AI 的 Prompt（见下文）
  const prompt = buildAgentPrompt(context, userInstruction)
  
  // 调用 AI（可对接多个 provider）
  const response = await callAI(prompt)
  
  // 解析 AI 返回，执行修改
  const result = await applyAgentEdit(response)
  
  res.json(result)
})

function buildAgentPrompt(context, instruction) {
  return `
你是 tang-slidex 的 HTML 编辑助手。slide 是一个 1280×720 的独立 HTML 文件。

## 当前 slide 文件
文件路径：${context.slideFile}

## 用户选中的区域（你需要修改的目标）
${context.fragments.map(f => `
- 标签：${f.tagName}.${f.classList}
  行号：${f.line}
  当前 HTML：
  \`\`\`html
  ${f.html}
  \`\`\`
`).join('\n')}

## 完整 slide HTML（参考上下文）
\`\`\`html
${context.fullSlideHtml}
\`\`\`

## 用户指令
${instruction}

## 要求
1. 只修改用户选中的区域（上面列出的元素）
2. 保持其他区域不变
3. 按照 tang-slidex 规范（1280×720 尺寸，CSS 变量等）
4. 返回 JSON 格式的修改指令（AgentEditResponse 格式）
5. 不要在 HTML 里引入外部 CSS/JS 库

## 输出格式（JSON）
返回合法的 JSON，格式如下：
\`\`\`json
{
  "editType": "replace-element",
  "elementReplacements": [
    {
      "line": 10,
      "file": "slides/slide-001.html",
      "newHtml": "<h1 class=\\"slide-title\\">新标题内容</h1>"
    }
  ],
  "explanation": "将标题修改为更有力的表达"
}
\`\`\`
`
```

#### 4.3.4 Agent 修改回写（行级替换）

```javascript
async function applyAgentEdit(agentResponse) {
  const { editType, patches, elementReplacements, newSlideHtml } = agentResponse

  if (editType === 'replace-slide') {
    // 整个 slide 替换
    const file = path.join(process.cwd(), agentResponse.slideFile)
    fs.writeFileSync(file, newSlideHtml, 'utf8')
    return { ok: true }
  }

  if (editType === 'replace-element') {
    // 按行号替换整个元素
    for (const replacement of elementReplacements) {
      const filePath = path.join(process.cwd(), replacement.file)
      let html = fs.readFileSync(filePath, 'utf8')
      
      // 找到目标行，替换整个元素（从开标签到对应闭合标签）
      html = replaceElementAtLine(html, replacement.line, replacement.newHtml)
      fs.writeFileSync(filePath, html, 'utf8')
    }
    return { ok: true }
  }

  if (editType === 'patch') {
    // 精确行级 patch
    const fileGroups = groupBy(patches, 'file')
    for (const [file, filePatches] of Object.entries(fileGroups)) {
      const filePath = path.join(process.cwd(), file)
      let html = fs.readFileSync(filePath, 'utf8')
      html = applyPatchesByLine(html, filePatches)
      fs.writeFileSync(filePath, html, 'utf8')
    }
    return { ok: true }
  }
}

/**
 * 找到 line 行的开标签，替换整个元素到对应闭合标签
 * 使用简单的标签栈追踪
 */
function replaceElementAtLine(html, line, newHtml) {
  const lines = html.split('\n')
  const lineContent = lines[line - 1]
  
  // 找到标签名
  const tagMatch = lineContent.match(/<([a-zA-Z][a-zA-Z0-9]*)/)
  if (!tagMatch) return html
  const tag = tagMatch[1]
  
  // 找到对应闭合标签（处理嵌套）
  let depth = 0
  let endLine = line - 1
  for (let i = line - 1; i < lines.length; i++) {
    const l = lines[i]
    depth += (l.match(new RegExp(`<${tag}[\\s>]`, 'g')) || []).length
    depth -= (l.match(new RegExp(`</${tag}>`, 'g')) || []).length
    if (depth === 0) { endLine = i; break }
  }
  
  // 替换 line-1 到 endLine 的内容
  lines.splice(line - 1, endLine - line + 2, newHtml)
  return lines.join('\n')
}
```

## 5. 两种路线的协同关系

两种路线**不是互斥的**，而是互补的，共享同一套基础设施：

```
┌─────────────────────────────────────────────────────────────────┐
│                      共享基础设施                                │
│                                                                  │
│  ① slideInspectorPlugin（Vite transform）                        │
│     → 给所有 slide HTML 元素注入 data-tang-line                  │
│                                                                  │
│  ② SelectionManager                                             │
│     → 管理当前选中的元素，生成 AgentEditContext                   │
│     → WYSIWYG 模式用它来追踪编辑目标                             │
│     → AI 模式用它来生成对话上下文                                │
│                                                                  │
│  ③ 行级回写引擎（applyPatchesByLine / replaceElementAtLine）     │
│     → WYSIWYG 的文字/样式修改走 applyPatchesByLine               │
│     → AI 的元素替换走 replaceElementAtLine                       │
└─────────────────────────────────────────────────────────────────┘
```

### 5.1 无缝模式切换设计

```
普通状态（浏览/演示）
  → 键盘 E：进入 WYSIWYG 编辑模式
  → 键盘 A（或点击 AI 图标）：进入 AI 对话模式

WYSIWYG 编辑模式
  → 双击文字：文本编辑
  → 拖拽：位置调整
  → 点击元素 + 按 A：切换到 AI 对话（元素自动进入选区上下文）

AI 对话模式
  → 点击选中元素（SelectionManager 工作）
  → 对话框输入指令
  → 对话完成后可继续 WYSIWYG 微调
```

### 5.2 Undo/Redo（两种模式统一）

所有修改都通过 `/api/save-slide` 或 `/api/ai-edit` 写回文件。利用 **git** 实现撤销：

```javascript
// 写回前自动 git add + git stash（可选）
// 或维护内存中的历史快照（简单方案）
const editHistory = new Map()  // file → html[]（最多保留 20 条）

function saveWithHistory(file, html) {
  if (!editHistory.has(file)) editHistory.set(file, [])
  const history = editHistory.get(file)
  history.push(fs.readFileSync(file, 'utf8'))  // 保存当前版本
  if (history.length > 20) history.shift()
  fs.writeFileSync(file, html, 'utf8')
}

// Ctrl+Z：POST /api/undo
server.middlewares.use('/api/undo', (req, res) => {
  const { file } = parseBody(req)
  const history = editHistory.get(file)
  if (!history?.length) { res.json({ ok: false, message: '没有更多撤销历史' }); return }
  const prev = history.pop()
  fs.writeFileSync(path.join(process.cwd(), file), prev, 'utf8')
  res.json({ ok: true })
})
```

## 7. 技术选型与实现路径

### 7.1 行号注入：HTML Parser 选型

| 选型 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 正则直接处理 | 零依赖，速度快 | 对复杂 HTML 边界情况不健壮 | ⭐⭐⭐ |
| `htmlparser2` | 轻量（6KB），保留格式，事件驱动 | 需要自行重建 HTML 字符串 | ⭐⭐⭐⭐⭐ |
| `node-html-parser` | API 友好，支持修改 | 序列化可能改变格式 | ⭐⭐⭐⭐ |
| `parse5` | W3C 标准，完整 | 重量级（200KB+） | ⭐⭐ |

**推荐：正则处理（MVP 阶段）+ htmlparser2（稳定阶段）**

### 7.2 AI 对接方案

| 方案 | 适合场景 | 实现方式 |
|------|---------|---------|
| **Kwaipilot API** | 内部集成，无缝 | IDE 内对话直接共享上下文 |
| **本地 ollama** | 离线使用 | POST ollama API |
| **OpenAI / Claude** | 通用场景 | 标准 API Key 配置 |
| **无 AI（纯手动）** | 降级/离线 | 路线一独立运行 |

**设计原则**：AI 接入层抽象为可插拔的 adapter，用户在 `package.json` 的 `tang-slidex` 配置项中指定：

```json
{
  "tang-slidex": {
    "ai": {
      "provider": "openai",   // "openai" | "anthropic" | "ollama" | "kwaipilot"
      "model": "gpt-4o",
      "baseUrl": "https://api.openai.com/v1"
    }
  }
}
```
